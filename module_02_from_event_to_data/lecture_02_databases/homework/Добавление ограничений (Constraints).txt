1.Ограничения NOT NULL для сущностей sales и products.

Таблица sales
sales_id не может быть пустым, потому что это первичный ключ таблицы.
transaction_number не может быть пустым, потому что это номер транзакции.
sales_timestamp не может быть пустым, потому что для каждой продажи важно знать дату и время совершения операции.
product_id не может быть пустым, потому что каждая продажа должна быть связана с конкретным продуктом.

Таблица products
product_id не может быть пустым, потому что это первичный ключ таблицы.
product_name не может быть пустым, потому что продукт должен иметь название.
price не может быть пустым, потому что для продажи товара необходимо знать его цену.
category_id не может быть пустым, потому что каждый продукт должен относиться к определённой категории

2.Укажите, какие ID должны быть уникальными (UNIQUE).

Таблица sales
sales_id должен быть уникальным, потому что это первичный ключ таблицы. Каждая продажа должна иметь свой уникальный идентификатор.
transaction_number также может быть уникальным, потому что в данном файле номер транзакции не повторяется и используется для идентификации операции продажи.

Таблица products
product_id должен быть уникальным, потому что это первичный ключ таблицы. Каждый продукт должен иметь свой уникальный идентификатор.

3.Предложите 2–3 ограничения CHECK (например, для числовых полей, которые не могут быть отрицательными, или для логических полей, которые могут принимать только два значения).

Таблица sales

Поле quantity показывает количество проданного товара. Оно не может быть отрицательным или равным нулю, потому что продажа должна содержать хотя бы одну единицу товара.
Поле discount показывает размер скидки. Скидка не может быть меньше 0.

Таблица products


Поле price показывает цену продукта. Цена товара не может быть отрицательной или равной нулю, потому что продукт должен иметь положительную стоимость.
